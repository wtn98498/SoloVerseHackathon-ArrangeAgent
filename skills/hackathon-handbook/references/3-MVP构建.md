# MVP 构建

把 idea 变成评委能看的 demo。黑客松的 demo 不是产品，是"那个记得住的瞬间"的可点击版本。最大的风险不是功能不够，而是贪多做不完、或做出来一股 AI 味没质感。

默认技术栈：**Next.js（App Router）+ Vercel 部署**。理由：和部署无缝、生态成熟、AI 写得好、前端表现力强。除非用户另有要求或完全不懂代码（这种情况见文末）。

## 第零步：先问赞助商技术（动手前必问）

黑客松评分常有一项是"有没有用主办方 / 赞助商的技术"——赏金猎人的打法就是每场专门研究赞助商在推什么，然后把它编进作品。所以写代码前先问用户（用 AskUserQuestion）：

- 这场比赛的**主办方 / 赞助商**是谁？有没有**指定或加分的技术栈**？（如某条公链、Supabase、某个 AI 模型/API、某 SDK）
- 有没有"**用了就加分**"的 feature 或单独的 bounty 赛道？

拿到答案后，把赞助商技术**映射到 demo 的某个真实功能上**，而不是硬塞个 logo——例如用赞助商的 AI API 承担核心生成、用 Supabase 做登录/存储、用某 SDK 做关键交互。如果用户不清楚，引导他去看比赛页面 / 赞助商列表；暂时查不到就先按通用栈做、预留接入点。

## 第一步：砍到 1–2 个场景（最重要）

动手前先收敛，这一步决定 demo 能不能做完。

- 从创意阶段拿到的叙事里，找出**那个"记得住的瞬间"**，只做它。
- **最多 1–2 个核心场景**。把所有"以后可以加"的功能列一个 "不做清单"，明确不做。
- 黑客松节奏参考：约 12 小时内做出 demo，并**全程不断砍功能、收敛 idea**。功能每多一个，质感和完成度就掉一截。
- 判断标准：这个 demo 能不能在 60–90 秒视频里讲完？讲不完就再砍。

主动帮用户做减法：列出他想做的所有功能，标出哪 1–2 个是 pitch 真正需要的，其余划掉。

## 第二步：前端为主，后端为辅

评委看的是体验，不是架构。把工程预算压倒性地投在前端体验上。

后端策略，按从省力到费力排序，优先选靠前的：
1. **纯前端 + 假数据 / 本地 state**：能用写死的数据演示，就不要碰后端。大多数 demo 这样就够。
2. **直接调 AI API**（OpenAI / 等）：AI 类 demo 的核心通常是一次模型调用，前端直接调、把 key 放环境变量即可（demo 阶段够用）。
3. **Supabase 托管**：需要登录 / 存数据 / 实时时再用。它能把"后端"压成几行客户端代码，且常是赞助商技术——用它还能拿关联性的分。
4. 自己写完整后端：除非比赛要求，否则黑客松里基本不值得。

提醒：不要为了"看起来专业"去搭一堆用不到的后端。省下的时间全部投到前端质感和 demo 上。

### 接 AI API（demo 几乎都要用，重点）

大多数 AI demo 的核心就是一两次模型调用。**动手前就把 API 配好**，别等到要演示时才发现没 key、没额度。带用户走一遍：

1. **选 provider**：优先用赞助商指定 / 加分的 AI（比赛在推 Anthropic、OpenAI 或某模型，就用那个，顺便拿关联性的分）；没指定就挑用户已经有 key 的。
2. **拿 key**：去对应平台控制台创建 API key（如 platform.openai.com、console.anthropic.com）。多数要绑卡 / 看额度——黑客松常有赞助商发的免费额度或赠金，优先用，问主办方要。
3. **配到环境变量，绝不写死在代码里**：项目根目录建 `.env.local`，写 `OPENAI_API_KEY=...`（或对应变量名）；确认 `.gitignore` 含 `.env*`，别把 key 推上 GitHub。
4. **用服务端调用保护 key**：在 Next.js 的 route handler（`app/api/.../route.ts`）里调 AI，把 key 留在服务端，不要在浏览器端暴露。
5. **上线时同步**：在 Vercel 项目 Settings → Environment Variables 里再填一遍同样的 key，然后 Redeploy（本地能跑不等于线上能跑）。

各 provider 的具体配置代码和最小可跑示例见 `references/ai-api-setup.md`，按用户选的 provider 套用。

## 第三步：用"反 AI 味"规则把前端做出质感

这是 demo 拉开差距的地方。AI 默认生成的前端往往千篇一律——居中大标题、紫蓝渐变、圆角卡片三件套、Inter 字体、一眼假。**动手写任何 UI 之前，先读 `references/anti-slop-frontend.md`，并逐条遵守里面的规则与发布前检查清单。**

一句话原则：先定一个明确的设计方向（受众、气质、参考），再写代码；不要让模型用默认审美自由发挥。

> 这套规则提炼自开源的 Taste Skill（tasteskill.dev）。想要持续更新的完整版，可以在自己的项目里装：`npx skills add Leonxlnx/taste-skill`。本篇已内置其核心，离线可用。

## 动手前：出一份计划，确认后再开发

把前面几步收敛成一份**简短计划**给用户，确认或调整后再写代码。计划包含：

- **场景**：做哪 1–2 个（以及明确不做的）。
- **技术栈 + 赞助商技术**：用了哪个赞助商技术、落在哪个具体功能上。
- **AI API**：用哪个 provider、key 准备好了没、额度从哪来。
- **后端策略**：假数据 / 直接调 API / Supabase。
- **页面与组件清单 + 时间预算**：大致几个页面、每块预计多久。

用户点头后再进入下面的搭建。别跳过这步——计划是帮用户在动手前就发现"key 没有""赞助商技术没接上""场景太大做不完"这些坑。

## 第四步：搭起来跑通

标准最小流程（按用户熟练度调整详略）：

```bash
npx create-next-app@latest my-demo   # 选 App Router、Tailwind
cd my-demo
npm run dev                           # 本地预览 http://localhost:3000
```

然后：
- 先把那 1–2 个场景的页面骨架搭出来，套用 anti-slop 规则。
- 接上假数据 / AI API / Supabase（按第二步选的）。
- 反复看真机效果，对着 demo 脚本调"那个瞬间"的呈现。
- 留出时间给部署和 demo 视频——别做到最后一秒。

## demo 做完后：主动引导部署（别忘了这步）

demo 在本地能跑还不算完——**大多数黑客松提交都要一个公开 live link，没上线等于白做。** 所以 demo 一做完，就主动把用户推向部署，不要等他问：

- 明确告诉用户："demo 跑通了，现在最关键的一步是把它部署上线，拿到一个评委能点开的链接。" 然后**直接衔接 阶段四·部署上线**（`references/4-部署上线.md`，Vercel 为主，不会 GitHub 也有保姆教程）。
- 别拖到最后一刻部署——上线常会暴露"本地能跑、线上 500"的问题（多半是环境变量没在 Vercel 配），留时间排查。
- 部署、拿到 `vercel.app` 链接后，再做这两件收尾的事：
  - **阶段五·路演PPT**（`references/5-路演PPT.md`）：做一版"完整版 deck"，把真实截图和线上链接放进去（先有链接，deck 的 demo 页和结尾页才能填上线地址）。
  - 用线上版本录 demo 视频，避免现场翻车。

一句话节奏：**做完 demo → 立刻部署拿链接 → 再做 PPT / 录视频 → 提交。**

## 如果用户完全不会写代码

完全可以打黑客松。给两条路：
- **用 AI 生成器直接出 demo**：Gemini Canvas、v0、Lovable、Bolt、Cursor 等，能让不懂编程的人做出黑客松级别的页面。把第三步的"反 AI 味"规则当成提示词喂给它们，质感会好很多。
- **不做代码也能赢**：把精力放在 idea、PPT、demo 视频、和赞助商碰想法、现场演讲上——这些对非开发者同样能拿奖。

无论哪条路，核心不变：**收敛场景、前端为主、把那个"记得住的瞬间"做出质感。**
